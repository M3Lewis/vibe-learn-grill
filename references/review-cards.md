
# 复习卡片参考

`LEARN-GRILL.md` 是唯一的持久化学习状态文件。它应该保持小巧且有用。

## 卡片创建条件

仅在用户出现以下情况时创建或更新卡片：

- 回答错误
- 答案模糊或凭记忆蒙对
- 混淆两个概念
- 无法解释数据流
- 无法预测边界情况
- 无法将想法迁移到相近的改动
- 遗漏项目中高频的核心概念

**不要**为每个事实都创建卡片。

## 卡片格式

```md
## CARD-001

Topic（主题）:
File（关联文件）:
Status（状态）: weak | shaky | stable | mastered
Strength（强度）: 1/5
Last reviewed（上次复习）:
Next review（下次复习）:

Question（问题）:

Ideal answer（理想答案）:

My mistake（我当时的错误/盲点）:

Review prompt（复习提示）:
```

> 注意：`Status` 字段建议使用英文 `weak/shaky/stable/mastered`，以便与调度规则中的表格对应。但你也可以在填写时使用中文“弱/不稳/稳定/掌握”，调度规则同样理解。

## 调度规则

使用简单的规则，而不是完整的间隔重复引擎：

| 结果 | 更新 |
|---|---|
| 回答错误 | `Status: weak`，`Strength: 1/5`，`Next review: next-session` |
| 回答不稳（半对半错） | `Status: shaky`，`Strength + 1`，`Next review: next-session` 或 tomorrow |
| 正确但较慢 | `Status: stable`，`Strength + 1`，`Next review: 3 days` |
| 正确且能举例/预测边界 | `Strength + 2`，`Next review: 7 days` |
| 连续两次答得又好又稳 | `Status: mastered`，之后只随机抽查 |

## 学习进度区块维护规则

`LEARN-GRILL.md` 顶部的“学习进度”区块用于跨会话恢复进度。AI 和用户共同维护，遵循以下规则：

### 新对话启动时

1. 加载 `LEARN-GRILL.md`，首先读取“学习进度”区块。
2. 根据区块内容决定行为：
   - 若“当前活跃模块”有未稳定的卡片（`weak`/`shaky`），**优先复习这些卡片**（不超过 5 张）。
   - 复习完毕后，主动询问：“当前活跃模块的卡片已复习完，是否继续学习‘下一模块’？还是再复习一些旧卡片？”
   - 若“当前活跃模块”为空（无未稳定卡片），则直接询问是否开始学习“下一模块”。

### 会话中的更新触发条件

- 用户明确说：“完成当前模块” / “标记 X 模块为完成” → AI 将该模块从“当前活跃模块”移动到“已完成的模块”，并记录完成日期。若用户已指定“下一模块”，则将其设为新的“当前活跃模块”；否则询问：“你想把哪个模块设为下一个？”
- 用户说：“暂停当前模块，切换到 Y 模块” → 更新“当前活跃模块”为 Y，原模块可移至“已完成的模块”或保留在注释中（用户决定）。
- 当某个模块的所有卡片状态都变成 `mastered` 时，AI **可选**询问：“模块 X 的所有卡片都已掌握，要把它标记为‘已完成’吗？”（不强制，避免每次打扰）
- 每次会话结束时，AI 应自动更新“上次会话结束时间”为当前时间（如果用户允许写入文件）。

### 区块格式模板（严格遵循）

```markdown
## 学习进度

- **已完成的模块**：模块名 (日期, 简单备注)
- **当前活跃模块**：模块名，尚有 X 张卡 weak/shaky
- **下一模块**：模块名（未开始 / 进行中）
- **上次会话结束时间**：YYYY-MM-DD HH:MM
```

用户可手动编辑该区块，AI 不得覆盖用户的手动修改（只追加或按指令更新）。

## 会话热身

每次编码/学习会话开始时：

1. 加载 `LEARN-GRILL.md`（若存在）。
2. 选择 3 张卡片，最多 5 张。
3. 优先选 `weak`、`shaky`、到期、以及当前模块相关的卡片。
4. 一次只问一张卡片的问题。
5. 用户回答后更新卡片状态（如果明确允许写入文件）。
6. 热身完成后继续构建/学习。

如果用户要求跳过复习，则跳过本次，但在会话结束前提示“还有 X 张弱卡片未复习”。

## 维护模块-卡片映射表

为了直观展示问题在哪些模块集中出现、模块间可能存在的关联，`LEARN-GRILL.md` 顶部可以包含一个可选的“模块-卡片索引”区块。

### 何时生成/更新映射表

- **主动触发**：用户明确要求“生成模块-卡片映射表”或“更新索引”。
- **自动触发（宽松条件）**：当卡片总数 ≥ 8 且当前会话中卡片数量发生变化（新增、状态更新）时，AI 可以询问用户：“卡片已积累到 X 张，需要我帮你生成/更新模块-卡片映射表吗？” 若用户同意，则执行。
- **避免频繁更新**：不要每次卡片变动都自动生成；建议每 5-10 张新卡或每完成一个里程碑（如某个模块的所有卡片状态变为 `stable/mastered`）再整理一次。

### 如何生成

1. 扫描所有卡片的 `File` 字段（或 `Topic` 中提取的模块名），按目录/模块分组。
2. 为每个模块列出关联的卡片编号（如 `CARD-001, CARD-003`）。
3. 用缩进或箭头表示模块间的依赖关系。AI 可以根据代码结构或卡片内容中的关联文件推测简单的依赖，但最终依赖关系以用户手动补充为准。AI 可以在生成后提醒用户：“你可以手动添加箭头（→）来表示依赖关系，例如 `- 配置模块 → 被路径模块依赖`。”
4. 将生成的索引区块放置在 `LEARN-GRILL.md` 顶部（`## 复习规则` 之前）。如果已有旧区块，则完全替换。

### 生成格式模板

```md
## 模块-卡片索引

- **模块A** (`路径/`) → CARD-001, CARD-002
  → （用户可补充依赖说明）
- **模块B** (`路径/`) → CARD-003
- **模块C** (`路径/`) → CARD-004, CARD-005
  → （用户可补充）
```

### 示例对话

用户：“帮我生成模块-卡片映射表”

AI：（扫描 `LEARN-GRILL.md`，生成索引，写入文件）

AI：“已生成索引。你可以手动在箭头后添加依赖关系说明，比如 `→ 被配置模块调用`。”
